# 측정 결과 리포트

sngrep 캡처 드롭 가설의 실측 결과. 환경·방법은 [README](README.md), 도구 사용법은
[bench/README.md](bench/README.md) 참고.

**측정 환경**: Ubuntu 26.04 VM (4 vCPU / 4GB), VirtualBox, 호스트 i5-1335U/16GB.
트래픽은 VM 루프백(`lo`), 발신/수신/캡처 모두 VM 내부. 측정일 2026-06-29.

> ⚠️ 노트북(15W) 발열 throttling으로 **절대 PPS 수치는 실행마다 변동**한다(정성적
> 결론은 영향 없음). 정밀 절대치가 필요하면 안정 데스크톱에서 재측정 필요.

## T1 — 드롭이 나는가, 어디서 나는가  ✅ 입증

SIPp로 콜레이트를 100→8000 cps로 올리며 `pcap_stats`를 샘플링.

- **드롭 발생**: 시그널링 부하에서 누적 **20~36% 손실**.
- **위치 = 커널 캡처 링버퍼**: 드롭이 전부 `ps_drop`이고 **`ps_ifdrop`=0** (전 구간).
  → NIC 한계가 아니라 **유저공간(sngrep)이 링버퍼를 못 비워서 나는 역압**. 문서의
  ②커널 캡처 링버퍼 시나리오 그대로.

### 버퍼를 키워도 안 막힌다 (T6)

| 링버퍼 `-B` | 누적 드롭% |
|---|---|
| 2 MB (기본) | 21.6% |
| 16 MB | 35.8% |
| 64 MB | 32.2% |

64MB(≈0.67초치 버퍼)에서도 32%가 **지속적으로** 빠졌다. 지속 과부하(평균 처리율 <
평균 도착율)에선 버퍼는 넘침을 **지연**시킬 뿐 막지 못한다. → 버퍼 확장만으론 불가,
**더 빨리 비우는 비동기/병렬 파이프라인 필요**.

> 위 표의 B2<B16<B64 역전은 발열 confound(버퍼가 크면 드롭은 줄어야 함). 정성적
> 결론("어떤 버퍼에서도 지속 드롭")은 견고.

### 자원(4vCPU/4GB) 탓이 아니다

붕괴 시점에 **`cpu_all` 최대 60~68%** (≈1.5코어 유휴), **RAM 최저 여유 ~1.9GB**
(OOM·스왑 없음). **코어가 놀고 메모리도 남는데 드롭** → 총 CPU 부족이 아니라
**단일 스레드 한계** (sngrep의 한 캡처+파싱 스레드가 코어 하나를 못 넘고, 남는 코어를
병렬화 못 해 못 씀). 코어를 더 줘도 현재 sngrep은 못 쓴다.

![capture benchmark](bench/results/20260629_045659_signaling_B2/report.png)

위: 드롭(빨강)은 2000cps 단계부터 시작(누적 17%로 상승). 아래 CPU 패널 — 총합(진초록)은
60%대에 머물고 코어별(연초록)만 100%를 찍는다 = 한 코어 포화, 나머지 유휴. RAM(주황)은
1.5GB로 여유.

## T4 — 그 단일 스레드는 어디서 시간을 쓰나  ✅ 파싱+그룹화

`parse_packet`에 단계별 타이머(`PROFILE=1`)를 심어 측정 (signaling 램프).

| 단계 | 비중 | 총 시간 | 평균/패킷 |
|---|---|---|---|
| **파싱 + 그룹화** | **90.0%** | 80.8 s | **104.4 µs** |
| 락 대기 (UI/표시 경합) | 8.2% | 7.3 s | 9.5 µs |
| IP 재조립 | 1.7% | 1.5 s | 2.0 µs |
| dump | 0.1% | 0.12 s | 0.18 µs |
| TCP 재조립 | 0% | 0 | — (UDP) |

- **병목은 압도적으로 파싱+그룹화(90%)** — 재조립·dump·락이 아니다. 문서가 미검증으로
  남긴 "병목이 파싱인가?"는 **YES**. → **파싱 워커풀(병렬화) 정당화**.
- 패킷당 ~112µs → **단일 스레드 천장 ~9,000 pps**. T1이 수만 pps에 곧장 무너진 이유와
  정확히 일치.
- 락 대기 8% → 표시 경합은 실재하나 부차적. "표시만 분리"로는 8%만 회수.

### 파싱 vs 그룹화 (parse+group 단계 내부 분해)

`sip_check_packet`에 파싱/그룹화 함수를 따로 계측:

| 구분 | 시간 | 단계(85.5s) 중 | 비고 |
|---|---|---|---|
| **SIP 파싱** | 62.2 s | **72.7%** | Call-ID·메서드·응답 추출이 **패킷마다 regex**(최대 10KB, ~74µs/패킷) |
| 그룹화 | 12.6 s | 14.8% | find/add/state/retrans (~15µs/패킷) |
| 기타(페이로드 복사·할당 등) | 10.7 s | 12.5% | |

→ **파싱이 압도적, 그룹화는 미미.** 설계 함의:
- **파싱은 패킷 독립적 → 워커풀로 병렬화** (가장 큰 이득).
- **그룹화는 단일 스레드로 충분** — 공유 상태(Call-ID 맵)지만 15%뿐이라 **Call-ID 샤딩
  (문서 4.5)은 불필요**.

![capture-thread time breakdown](bench/results/20260629_073854_signaling_B2/profile.png)

위: 스택 영역에서 빨강(parse+group)이 캡처 스레드 시간을 거의 다 차지하고, 고레이트
구간에선 250ms 간격당 ~250ms로 꽉 차 단일 코어가 포화된다. 아래 막대 = 단계별 비중·평균.

## 종합 결론

1. sngrep은 고부하에서 **커널 캡처 링버퍼에서 드롭**한다 (`ps_drop`, ifdrop=0).
2. **버퍼 확장으론 못 고친다** — 지속 과부하.
3. 원인은 자원 부족이 아니라 **단일 스레드 직렬 처리**.
4. 그 스레드 시간의 **90%가 파싱+그룹화** → 여기를 병렬화/분리하는 것이 핵심.

→ 문서의 새 아키텍처(캡처/파싱 분리 + 파싱 워커풀) 전제가 데이터로 섰다.

## 남은 것

- **파싱 vs 그룹화 분리 계측** (`sip.c`: sip_check_packet vs call_add_message) — 90%를
  둘로 쪼개 "파싱만 병렬화 vs 그룹화 샤딩도 필요" 결정.
- **RTP/RTCP 차원** 측정 (현재 시그널링만; `SCENARIO=rtp`).
- 발열 통제한 환경에서 정밀 붕괴 PPS 곡선.
